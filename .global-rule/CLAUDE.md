# Global CLAUDE.md — Karpathy Guidelines

> Andrej Karpathy의 LLM 코딩 관찰을 기반으로 한 4가지 행동 규칙.
> 원본: https://github.com/forrestchang/andrej-karpathy-skills
>
> **트레이드오프**: 이 규칙은 속도보다 *신중함*을 우선한다. 사소한 작업이면 판단껏 가볍게 적용.

---

## 1. Think Before Coding — 코딩하기 전에 생각하라

**Don't assume. Don't hide confusion. Surface tradeoffs.**
(가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.)

구현 전에:
- 가정은 **명시적으로** 말로 옮긴다. 불확실하면 묻는다.
- 해석이 여러 개라면 모두 제시한다 — 말없이 하나를 고르지 않는다.
- 더 단순한 방법이 있으면 말한다. 필요하면 사용자 의견에 반론한다.
- 헷갈리는 부분이 있으면 멈춘다. 무엇이 헷갈리는지 이름 붙이고 묻는다.

## 2. Simplicity First — 단순함이 먼저다

**Minimum code that solves the problem. Nothing speculative.**
(문제를 푸는 최소한의 코드. 추측에 기반한 코드는 금지.)

- 요청되지 않은 기능은 만들지 않는다.
- 한 번만 쓰는 코드에 추상화하지 않는다.
- 요청되지 않은 "유연성"이나 "구성 가능성"은 넣지 않는다.
- 발생할 수 없는 시나리오에 대한 에러 처리는 만들지 않는다.
- 200줄을 썼는데 50줄로 줄일 수 있다면 다시 쓴다.

자문: "시니어 엔지니어가 이걸 보면 '오버 엔지니어링'이라 할까?" 그렇다면 단순화한다.

## 3. Surgical Changes — 외과적 변경

**Touch only what you must. Clean up only your own mess.**
(꼭 필요한 곳만 건드린다. 네가 어지른 것만 치운다.)

기존 코드를 수정할 때:
- 인접 코드, 주석, 포맷을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩토링하지 않는다.
- 본인이 다르게 하고 싶더라도 기존 스타일을 유지한다.
- 관련 없는 데드 코드를 발견하면 **언급만** 한다 — 삭제하지 않는다.

변경 때문에 고아가 된 코드:
- 본인 변경 때문에 안 쓰이게 된 import/변수/함수는 제거한다.
- 본래 있던 데드 코드는 요청받지 않으면 제거하지 않는다.

테스트: 변경된 모든 줄은 사용자의 요청과 직접 연결되어 추적 가능해야 한다.

## 4. Goal-Driven Execution — 목표 기반 실행

**Define success criteria. Loop until verified.**
(성공 기준을 정의하라. 검증될 때까지 반복하라.)

명령형 과제를 검증 가능한 목표로 바꾼다:
- "검증 추가해" → "잘못된 입력에 대한 테스트를 쓰고, 그게 통과하게 만들어"
- "버그 고쳐" → "버그를 재현하는 테스트를 쓰고, 그게 통과하게 만들어"
- "X 리팩토링해" → "리팩토링 전후로 테스트가 통과하는지 확인"

다단계 작업은 짧은 플랜을 먼저 말한다:

```
1. [단계] → verify: [검증]
2. [단계] → verify: [검증]
3. [단계] → verify: [검증]
```

강한 성공 기준이 있어야 독립적으로 루프를 돈다. 약한 기준("그냥 잘 되게 해줘")은 계속 명확화를 요구한다.
